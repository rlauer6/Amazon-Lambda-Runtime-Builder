# Amazon::Lambda::Runtime::Builder 1.1.0

Released: Sunday, June 14, 2026

## Summary

This release removes the need for a per-project `builder/` directory.
The build framework - `Makefile`, all `.mk` includes, and the Dockerfile
- is now installed once with the distribution and referenced from any
project via `File::ShareDir`. A project needs only three files:
`Makefile.builder` (a three-line shim), `policies`, and a handler
module. Project-specific configuration (function name, handler class,
trigger settings, etc.) lives in a single `lambda.env` file at the
project root.

This is the first version validated end-to-end against a real,
previously-working deployment using `lambda.env` - see "Validation"
below.

## Highlights

### New: immutable, shared framework via `Makefile.builder`

`share/Makefile` is renamed to `share/Makefile.mk` and is no longer
copied into projects. A new three-line `share/Makefile.builder` is the
only build-related file a project keeps locally:

```makefile
#-*- mode: makefile; -*-
# Makefile.builder - generated by `alr-builder install`, safe to regenerate

FRAMEWORK_DIR := $(shell perl -MFile::ShareDir=dist_dir -e 'print dist_dir("Amazon-Lambda-Runtime-Builder")')
include $(FRAMEWORK_DIR)/Makefile.mk
```

Because the real Makefile is included via `-f Makefile.builder` rather
than being named `Makefile`, it does not collide with a project's own
`Makefile` (e.g. the one generated by `CPAN::Maker`/`Makefile.PL` for
`make dist`/`make test`).

Inside `Makefile.mk`, every include of a trigger or helper makefile is
now resolved relative to `$(FRAMEWORK_DIR)`:

```
include $(FRAMEWORK_DIR)/help.mk
include $(FRAMEWORK_DIR)/ecr-create-repo.mk
include $(FRAMEWORK_DIR)/ecr-login.mk
include $(FRAMEWORK_DIR)/sns.mk
include $(FRAMEWORK_DIR)/sqs.mk
include $(FRAMEWORK_DIR)/eventbridge.mk
include $(FRAMEWORK_DIR)/s3.mk
include $(FRAMEWORK_DIR)/streaming.mk
```

The previous `.includes/` subdirectory is gone; these files are flat in
`share/`. `INCLUDES_DIR` is removed as it is no longer used.

### New: `lambda.env` replaces `project.mk`

`Makefile.mk` now does `-include lambda.env` instead of `-include
project.mk`. `lambda.env` is a flat `KEY = value` file at the project
root holding every project-specific variable - `FUNCTION_NAME`,
`HANDLER_CLASS`, `REPO_NAME`, `ROLE_NAME`, `QUEUE_NAME`, `BATCH_SIZE`,
`VISIBILITY_TIMEOUT`, `RETENTION`, `RECEIVE_COUNT`, `DLQ_NAME`,
`DLQ_RETENTION`, `BUCKET_NAME`, `KEY_PREFIX`, `S3_EVENT`, `TIMEOUT`,
`MEMORY`, `CONCURRENCY`, and so on. It is hand-writable; no generator is
required for this release.

### `docker/` directory eliminated

The `cpanfile` and `debian-packages` files generated from the
distribution tarball's `META.json` are now written to `$(CACHE_DIR)`
instead of `$(BUILDER_HOME)/docker/`. The `image` target assembles a
temporary build context with `mktemp -d`, copying
`$(FRAMEWORK_DIR)/Dockerfile`, `$(CACHE_DIR)/cpanfile`, and
`$(DIST_TARBALL)` into it before calling `docker build`. The temporary
directory is removed via `trap ... EXIT`. No files are written into or
read from a `docker/` subdirectory anywhere in the build.

### Fixed: `DIST_TARBALL` location

`DIST_TARBALL` now globs `$(BUILDER_HOME)/*.tar.gz` (the project root).
Previously it globbed `$(BUILDER_HOME)/../*.tar.gz`, which assumed
`make` was run from a `builder/` subdirectory one level below the
tarball. The corresponding error message now reads `No tarball found in
$(BUILDER_HOME) - run 'make dist' first`.

### `HANDLER_CLASS` is now required

```makefile
ifeq ($(HANDLER_CLASS),)
  $(error You must specify the HANDLER_CLASS)
endif
```

The previous silent default of `HANDLER_CLASS ?= LambdaHandler.pm` is
removed. `HANDLER_CLASS` must be set via `lambda.env` or on the command
line.

### Removed: `REINSTALL_PACKAGES`

`REINSTALL_PACKAGES` was passed to `docker build` as a `--build-arg`,
but `share/Dockerfile` has no corresponding `ARG REINSTALL_PACKAGES` and
never referenced it - Docker silently ignored it. The variable, its
`--build-arg`, and its documentation comment have been removed entirely.
The existing `DARKPAN_REBUILD` cache-bust combined with the final `RUN
cpm install ${RESOLVER}` step already re-resolves all dependencies
against the DarkPAN on a forced rebuild, making a separate
per-module-reinstall mechanism unnecessary.

### `DIST_TARBALL` no longer required for all targets

```makefile
NEEDS_TARBALL := $(filter-out clean,$(MAKECMDGOALS))
...
ifneq ($(NEEDS_TARBALL),)
ifeq ($(DIST_TARBALL),)
  $(error No tarball found in $(BUILDER_HOME) - run 'make dist' first)
endif
endif
```

`make clean` (and any other target added to the filter-out list in the
future) no longer fails with a missing-tarball error when no tarball is
present.

### Two shell bugs fixed in `deploy` and `lambda-function`

- In `deploy`, the `chmod -f 644 $@ 2>/dev/null || true` step was
  joined to the following `$(call ecr_login,...)` line with a trailing
  `\`. Because `ecr_login`'s own expansion begins with `$(NO_ECHO)`
  (`@`), that `@` ended up mid-shell-command rather than at the start of
  a recipe line, producing `/bin/bash: ... @URI=...: No such file or
  directory`. The `chmod` step is now its own recipe line.
- In `lambda-function`, `if [[ "$$function" = "" }]]` (a stray `}`) is
  fixed to `if [[ "$$function" = "" ]]`.

### `MANIFEST.json` reduced to three files

```json
{
    "files": {
        "LambdaHandler.pm" : "LambdaHandler.pm",
        "policies" : "policies",
        "Makefile.builder": "Makefile.builder"
    }
}
```

Previously this listed fourteen files (`Dockerfile`, `Makefile`, every
`.includes/*.mk`, `.gitignore`, both payload JSON files, `README.md`,
`project.mk`). All of those now live only in the framework's `share/`
directory and are referenced via `$(FRAMEWORK_DIR)`; `alr-builder
install` copies only the three files above into a project.

### `sqs.mk`: new `DLQ_RETENTION` variable

```makefile
DLQ_RETENTION ?= 1209600
```

The dead-letter queue's `create-queue` call now uses
`retention:$(DLQ_RETENTION)` instead of a hardcoded `1209600`, so DLQ
message retention can be set per project via `lambda.env`.

### `Role::Install::cmd_install`: warn on copy failure

`copy($src, $dest)` is now checked; a failed copy produces `warn
"WARNING: could not copy $src to $install_dir/$dest\n"` instead of
failing silently.

### `buildspec.yml`

`extra-files` updated to package `share/Makefile.builder`,
`share/Makefile.mk`, and `share/help.mk`, replacing the old
`share/Makefile` entry.

### `project.mk` relocated

`share/project.mk` is renamed to `project.mk` at the distribution root
(100% content match, no changes). This is this distribution's own
`CPAN::Maker` project configuration and was never part of the installed
per-project scaffold; it had been misplaced under `share/`.

## Validation

This release was tested against the live `lambda-handler` function
(`OrePAN2-S3-Handler`, the S3-to-SQS DarkPAN indexer):

- `lambda.env` was hand-written for the project (`HANDLER_CLASS`,
  `REPO_NAME`, `FUNCTION_NAME`, `ROLE_NAME`, `QUEUE_NAME`, `BATCH_SIZE`,
  `VISIBILITY_TIMEOUT`, `RETENTION`, `RECEIVE_COUNT`, `DLQ_NAME`,
  `DLQ_RETENTION`, `BUCKET_NAME`, `KEY_PREFIX`, `S3_EVENT`, `TIMEOUT`,
  `MEMORY`, `CONCURRENCY`).
- `FRAMEWORK_DIR` resolved correctly via the real, installed
  `File::ShareDir::dist_dir("Amazon-Lambda-Runtime-Builder")` -
  `/home/rlauer/lib/perl5/auto/share/dist/Amazon-Lambda-Runtime-Builder`.
- `make -f Makefile.builder update-function` ran the full chain
  (`ecr-repo` -> `cpanfile`/`debian-packages`/`tarball-validated` ->
  `image` -> `deploy` -> `image-digest` -> `update-function`) and
  succeeded: Docker build completed, image pushed to ECR, and
  `UpdateFunctionCode` returned `"State": "Active"`.
- The resulting image digest (`sha256:20063e0e29b7...`) was identical to
  the digest already in ECR from the previous day's deployment built
  with the pre-1.1.0 framework - the new build machinery produces the
  same image for the same inputs.
- `make -f Makefile.builder clean` ran without error.
- A post-deploy `lambda-explorer lambda-handler` check showed the
  function `Active`, the SQS trigger `Enabled` with `batch=1`, the S3
  bucket notification correctly configured, and the DLQ empty. One
  pre-existing finding (`NO_PARTIAL_BATCH_RESPONSE`, unrelated to this
  release) remained.

## Upgrading

Existing projects using a `builder/` subdirectory with copied
`Makefile`/`.includes/*.mk`/`Dockerfile` should migrate to:

1. Remove the `builder/` directory.
2. Add `Makefile.builder` (above) and a `lambda.env` at the project
   root, setting at minimum `HANDLER_CLASS` and any variables
   previously set in `project.mk`.
3. Ensure the project's distribution tarball is built into the project
   root (`$(BUILDER_HOME)/*.tar.gz`), not a parent directory.
4. Run `make -f Makefile.builder <target>` in place of `make <target>`.

`REINSTALL_PACKAGES`, if set in a project's old `project.mk`, can be
removed - it has no effect.
