# Amazon::Lambda::Runtime::Builder 1.1.1

Released: Monday, June 15, 2026

## Summary

This release adds `builder.mk`/`builder.stub`, the mechanism by which a
project's own `project.mk` opts into ALRB's build/deploy targets
(`deploy`, `update-function`, `lambda-clean`, `lambda-sqs-pipeline`) without
copying any ALRB code into the project. `alr-builder install` is updated
to install this integration idempotently, and `MANIFEST.json` gains
per-file overwrite policies so user-customized templates
(`LambdaHandler.pm`, `policies`) are never silently clobbered while
framework files (`Makefile.builder`) are always refreshed.

## Highlights

### New: `builder.mk` / `builder.stub` - project integration

`share/builder.mk` provides four targets that delegate to
`Makefile.builder` via `$(MAKE) -f`:

```makefile
ALR_BUILDER = Makefile.builder

lambda-clean:
	$(MAKE) -f $(ALR_BUILDER) clean

deploy: $(TARBALL)
	$(MAKE) -f $(ALR_BUILDER) deploy

update-function: deploy
	$(MAKE) -f $(ALR_BUILDER) update-function

lambda-sqs-pipeline:
	$(MAKE) -f $(ALR_BUILDER) lambda-sqs-pipeline
```

`lambda-clean:` remove items from .cache
`deploy: $(TARBALL)` depends on `$(TARBALL)` if the project's own build
system (e.g. `CPAN::Maker::Bootstrapper`'s `project.mk`) defines it with
real file-dependency rules, giving `deploy`/`update-function` a
freshness check that rebuilds the distribution tarball if source has
changed. If `$(TARBALL)` is undefined, this prerequisite is a no-op;
ALRB's own `DIST_TARBALL`/`NEEDS_TARBALL` check in `Makefile.mk` still
requires at least one tarball to exist before any non-`clean` target
runs, independent of `$(TARBALL)`.

`share/builder.stub` is the snippet `alr-builder install` adds to a
project's `project.mk`:

```makefile
# >>> ALRB builder.mk integration (managed by `alr-builder install`) >>>
ALRB_FRAMEWORK_DIR := $(shell perl -MFile::ShareDir=dist_dir -e 'print dist_dir("Amazon-Lambda-Runtime-Builder")' 2>/dev/null)

ifeq ($(ALRB_FRAMEWORK_DIR),)
  $(error Amazon::Lambda::Runtime::Builder is not installed - run: cpanm Amazon::Lambda::Runtime::Builder)
endif

include $(ALRB_FRAMEWORK_DIR)/builder.mk
# <<< ALRB builder.mk integration <<<
```

Like `Makefile.builder`, this resolves `ALRB_FRAMEWORK_DIR` fresh at
`make`-time via `File::ShareDir`, so the stub is machine-independent and
safe to commit. `builder.mk` itself is never copied into a project -
only this nine-line pointer is.

### `Makefile.builder`: same "is ALRB installed" check as `builder.stub`

```makefile
#-*- mode: makefile; -*-
# Makefile.builder - generated by `alr-builder install`, safe to regenerate, safe to commit
FRAMEWORK_DIR := $(shell perl -MFile::ShareDir=dist_dir -e 'print dist_dir("Amazon-Lambda-Runtime-Builder")' 2>/dev/null)

ifeq ($(FRAMEWORK_DIR),)
  $(error Amazon::Lambda::Runtime::Builder is not installed - run: cpanm Amazon::Lambda::Runtime::Builder)
endif

include $(FRAMEWORK_DIR)/Makefile.mk
```

A fresh checkout on a machine without ALRB installed now fails with a
clear, actionable message regardless of which entry point is used first
- `make` (via `project.mk`'s `builder.stub`) or
`make -f Makefile.builder` directly.

### `MANIFEST.json`: per-file overwrite policy

```json
{
    "files": {
        "LambdaHandler.pm" : { "dest" : "LambdaHandler.pm", "overwrite": "never" },
        "policies" : { "dest": "policies", "overwrite": "never" },
        "Makefile.builder": { "dest": "Makefile.builder", "overwrite": "always" }
    }
}
```

`LambdaHandler.pm` and `policies` are starting-point templates a user
customizes; `overwrite: never` means `alr-builder install` leaves an
existing file alone (no `--force` override - if a reset to the sample is
genuinely wanted, delete the file first and reinstall).
`Makefile.builder` has no project-specific content, so
`overwrite: always` refreshes it unconditionally on every install,
keeping it current with the installed ALRB version. Bare-string manifest
entries (none currently) retain the prior `--force`-gated behavior.

### `cmd_install`: idempotent `project.mk` integration

If `project.mk` doesn't exist, it's created from `builder.stub`. If it
exists and already contains the `# >>> ALRB builder.mk integration`
marker, `cmd_install` logs that it's skipping and leaves `project.mk`
untouched. Otherwise, `builder.stub` is appended, with the prior
`project.mk` backed up to `project.mk.bak` first. Re-running
`alr-builder install` is now a no-op for `project.mk` once the
integration is present.

`Install.pm` now imports `CLI::Simple::Utils qw(slurp)` - a missing
import in 1.1.0 that would have caused `cmd_install` to die with
"Undefined subroutine &...::Role::Install::slurp called" the first time
the `project.mk`-handling code ran.

### New CLI options: `--force`, `--profile`

`-f|--force` is read by `cmd_install`'s overwrite-policy check for any
manifest entry that isn't `always` or `never` (no current entries use
this middle tier).

`-p|--profile` is intended to set `AWS_PROFILE` for the process. As
implemented, `local $ENV{AWS_PROFILE} = $self->get_profile` is scoped to
`init()`, which returns immediately - the env var reverts before any
`cmd_*` method runs and constructs AWS clients. **`--profile` currently
has no effect.** Likely fix: remove `local` so the assignment persists
for the process lifetime.

### `buildspec.yml`

`extra-files` updated to package `share/builder.stub` and
`share/builder.mk`.

## Upgrading

Existing projects on 1.1.0 should run `alr-builder install` to pick up
the `project.mk` integration (`builder.mk`'s `deploy`/`update-function`/
`lambda-clean`/`lambda-sqs-pipeline` targets) and the refreshed
`Makefile.builder` with the ALRB-installed check. `LambdaHandler.pm` and
`policies`, if already customized, are left untouched.

Do not pass `--profile` until the `local` scoping issue above is fixed;
it will silently have no effect.
